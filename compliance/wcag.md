# WCAG 2.2 AA Accessibility

## Goal

The Paradigm Studio meets **WCAG 2.2 Level AA** for every interactive surface. Beyond regulatory compliance (the EU Accessibility Act of 2025, Section 508 in the US, and Article 50(5) of the EU AI Act), an accessible Studio is a strict superset of a usable Studio for everyone — keyboard navigation, color contrast, and predictable focus benefit power users as well as users with disabilities.

## Standard

WCAG 2.2 publishes 86 success criteria across four principles: Perceivable, Operable, Understandable, Robust ("POUR"). Level AA encompasses 55 of those criteria. Paradigm targets all 55 plus a small set of AAA criteria where the cost is low.

The 9 new criteria added in WCAG 2.2 (over 2.1) are all in scope:
- 2.4.11 Focus Not Obscured (Minimum) — AA
- 2.4.12 Focus Not Obscured (Enhanced) — AAA
- 2.4.13 Focus Appearance — AAA
- 2.5.7 Dragging Movements — AA
- 2.5.8 Target Size (Minimum) — AA
- 3.2.6 Consistent Help — A
- 3.3.7 Redundant Entry — A
- 3.3.8 Accessible Authentication (Minimum) — AA
- 3.3.9 Accessible Authentication (Enhanced) — AAA

All AA criteria are met; AAA criteria 2.4.12, 2.4.13, and 3.3.9 are also met because the cost is small.

## How accessibility is engineered

### Architecture

- **Radix UI primitives** for every interactive component (`@radix-ui/*`). Radix is built specifically for accessible primitives and handles ARIA attributes, focus management, and keyboard interactions correctly.
- **Headless logic + Tailwind for visuals.** This separates accessibility from styling, so a visual change cannot accidentally regress accessibility.
- **No third-party widgets** without an accessibility audit. Anything pulled from npm goes through a checklist: ARIA roles correct, keyboard navigable, screen-reader-tested.

### Color and contrast

- All text-on-background pairs satisfy **4.5:1** contrast for normal text and **3:1** for large text (1.4.3 AA).
- Non-text UI elements (icons, focus rings, borders) satisfy **3:1** (1.4.11 AA).
- The dark theme and light theme are tested independently.
- Color is never the *only* signal. Errors are red **and** carry an icon **and** carry text. (1.4.1 A)
- A high-contrast theme is offered as a user setting; it pushes all pairs above 7:1.

A custom CI check uses `pa11y` to scan storybook stories for contrast violations on every PR.

### Keyboard

- Every interactive element is reachable by Tab, in a logical order. (2.1.1 A, 2.4.3 A)
- No keyboard traps. (2.1.2 A) The only modal traps are confirmation dialogs, and they are properly contained with focus return.
- Visible focus indicator on every focusable element with **>= 3:1** contrast against its background (2.4.7 AA, 2.4.11 AA, 2.4.13 AAA). Indicators are at least 2px solid and adapt color to the element behind them.
- Keyboard shortcuts are documented in `?` overlay and customizable by the user. No shortcut uses single-letter keys without a modifier (2.1.4 A).
- Focus is never trapped behind a sticky toolbar (2.4.11 AA). All scrollable containers compute focus visibility.

### Screen readers

- Every page has a meaningful `<title>` and a single `<h1>`. (2.4.2 A, 1.3.1 A)
- Headings descend in logical order; no skipping levels.
- Form fields have associated labels via `<label for>` or `aria-labelledby`. Placeholder text is never the only label.
- Live regions for asynchronous status: agent thinking, save success, error toasts. (4.1.3 AA)
- Images carry meaningful `alt` text generated by the agent (or `alt=""` for purely decorative). (1.1.1 A)
- The 3D viewport (Three.js canvas) is exposed to screen readers via an `aria-label` describing the current scene plus a parallel DOM tree of the seed gene structure. Users can navigate the gene tree by keyboard as an alternative to the canvas.

Tested with NVDA + Firefox, JAWS + Chrome, VoiceOver + Safari, and Orca + Firefox on every release.

### Touch and pointer

- Target size minimum **24×24 CSS px** (2.5.8 AA), with the recommended **44×44** for primary actions.
- No essential function requires drag-only interaction (2.5.7 AA). Drag operations have a tap-to-pick + tap-to-place keyboard equivalent.
- Multi-finger gestures are not the only way to perform any action.
- Hover-only interactions are paired with click/focus equivalents.

### Motion

- The agent's "thinking" animation respects `prefers-reduced-motion`. (2.3.3 AAA — we hit this even though it's AAA.)
- Auto-playing video previews can be paused. (2.2.2 A)
- Carousels and animations meet **5 second** auto-stop or have visible controls. (2.2.2 A)
- No flashing content above 3 Hz. (2.3.1 A)

### Cognitive accessibility

- Error messages are specific, actionable, and offered with suggestions. ("Couldn't save: name is too long. The maximum is 80 characters.") (3.3.3 AA)
- Form inputs preserve user data on validation failure. (3.3.4 AA, 3.3.7 A)
- Authentication does not rely on cognitive function tests like CAPTCHA (3.3.8 AA, 3.3.9 AAA). Paradigm uses passkeys (WebAuthn) as the primary auth method, and they require no memorization.
- Help is consistently located in the top-right corner across all pages. (3.2.6 A)
- The agent disclosure ("you are talking to an AI") is plain language and visible from the start.

### Forms

- Required fields are marked with text, not just an asterisk.
- Field error messages appear inline next to the field, not at the top of the page.
- Field hints (`aria-describedby`) appear before the user makes mistakes, not only after.
- Tab order matches visual order.
- The Submit button is always reachable; long forms have an "anchor" sticky bar.

### Internationalization

- The Studio is built for translation: every string lives in `i18n/<locale>.json`, no hard-coded strings in components.
- Text expansion (German strings can be 30% longer than English) is tested on every release.
- Right-to-left languages (Arabic, Hebrew) flip the layout via `dir="rtl"` and CSS logical properties.
- Locale affects formatting (dates, numbers, currencies, plurals).

## Testing

| Layer | Tool | Frequency |
|---|---|---|
| Unit | `@testing-library/react` accessibility queries | Every PR |
| Component | Storybook + `@storybook/addon-a11y` (axe-core) | Every PR |
| Integration | `pa11y-ci` against the dev server | Every PR |
| End-to-end | Playwright with `@axe-core/playwright` | Every PR |
| Manual screen reader | NVDA / VoiceOver / JAWS / Orca | Each major release |
| Keyboard-only walkthrough | One engineer per release | Each major release |
| External audit | Third-party WCAG audit | Annually |

A PR cannot merge if axe-core reports any critical or serious issue.

## Audit findings tracking

Findings from external audits are tracked as GitHub issues with the label `accessibility`. Severity follows the WAI-ARIA categories:

- **Critical:** blocks a user from completing a primary task.
- **Serious:** significantly degrades experience.
- **Moderate:** noticeable issue but workarounds exist.
- **Minor:** cosmetic.

Critical and Serious must be fixed within the same release cycle they are reported. Moderate within two cycles. Minor at engineering discretion.

## Studio-specific challenges

The Studio is a **creative tool with a 3D viewport**, which is the hardest type of UI to make accessible. The strategies we apply:

1. **Parallel DOM tree.** The seed's gene structure is rendered as a DOM tree alongside the 3D viewport. Screen reader users can navigate the gene tree to understand the seed's structure even if they cannot see the viewport.
2. **Semantic descriptions.** The agent can verbalize a seed at the user's request: "This is a melancholy bard sprite. Tall, slender, holding a wooden harp. The color palette is muted blues and grays. The animation includes idle, walk, and a sad hum loop."
3. **Keyboard shortcuts for every viewport action.** Rotate, pan, zoom, select gene, modify gene — all reachable from the keyboard.
4. **Audio mode.** A toggleable mode that sonifies the gene tree (each gene type has a distinctive tone) so audio-only users can build a mental model.

## Accessibility statement

A public accessibility statement at `paradigm.app/accessibility` declares conformance, lists known limitations, provides contact information for accessibility issues, and links to the most recent third-party audit report. This statement is required by the EU Accessibility Act for digital services.
