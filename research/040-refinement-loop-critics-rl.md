# 040 — Refinement loop with critic models and RL from preferences

## Question
What are the critic models the GSPL evolution stack uses, how are they trained, and how does RL-from-preferences feed user signals back into the critics so they get better over time?

## Why it matters
Critics are evolution's compass. Without good critics, MAP-Elites fills cells with garbage and POET co-evolves into nonsense. With good critics, the same algorithms find the small fraction of seeds that are *actually good*. Critics are also the lever for personalization (Brief 034). They are the highest-leverage component of the entire evolution stack.

## What we know from the spec
- Brief 011 introduced critics as Layer 2 of the reliability stack (semantic checks).
- Brief 034 referenced critics in the self-improving agent loop.
- Brief 035 mentioned critics in the per-engine presets.

## Findings — the critic suite

### Three classes of critics

GSPL's critic suite has three classes per engine:

#### Class 1: Rule-based critics

Cheap, deterministic, interpretable. Encode hard constraints and well-known quality heuristics.

**Examples per engine:**
- **SpriteEngine**: silhouette readability score (background subtraction + perimeter ratio); palette harmony score (OKLab distance ratios); edge sharpness; alpha cleanliness.
- **MusicEngine**: key consistency; voice-leading violations count; rhythm density variance; cadence frequency.
- **GameEngine**: reachability of win condition; mean session length; control density; difficulty curve smoothness.
- **UIEngine**: contrast compliance; touch target coverage; focus order validity (already enforced as hard gates in Brief 026).

These critics produce a number in `[0,1]` and are pure functions of the seed/IR.

#### Class 2: Learned critics

Small models trained on labeled data. Capture the "looks/sounds/feels good" signal that rules can't.

**Architecture:** Small CNN/transformer/MLP per engine, ≤ 50M params. Inputs are the rendered output (or a feature extraction of it). Outputs are scalar quality scores.

**Training data sources:**
1. **Public exemplars** scraped or licensed at v1 to bootstrap.
2. **User accept/reject signals** from the exemplar feedback loop (Brief 031).
3. **Synthetic preference pairs** generated by perturbing high-quality seeds along sloppy FIM directions (Brief 019).

**Update cadence:**
- v1: pre-trained, no updates after release.
- v1.5: monthly retraining cycles incorporating user data (opt-in).
- v2: continual learning with a held-out validation set per release.

#### Class 3: Preference-based critics (RLHF-style)

Trained directly on user pairwise preferences ("A is better than B for me"). Uses a Bradley-Terry model or a small reward model.

**Why this is powerful:** Pairwise preferences are easier to collect than absolute scores, more reliable, and capture user-specific quality.

**Where the signals come from:**
- The studio's "compare two candidates" UI (Brief 050).
- Implicit pairs from the exemplar loop: an accepted seed beats a rejected seed in the same session.

**Per-user vs global:**
- A *per-user* preference critic is part of the self-improvement loop (Brief 034 — the critic ensemble).
- A *global* preference critic is trained from anonymized aggregate data (opt-in only) and shipped with releases.

### How they compose into a refinement loop

```
Generation N candidates
        │
        ▼
Rule-based critics (cheap, run on all)
        │
        ▼
Top-K survive (e.g., top 50%)
        │
        ▼
Learned critics (moderate cost)
        │
        ▼
Top-K' survive (e.g., top 25%)
        │
        ▼
Preference critics (most personalized)
        │
        ▼
Final ranking → next generation parents
```

This **early-out cascade** is the key cost optimization. Cheap critics filter out the obvious losers; expensive critics evaluate only the survivors.

### Critic ensemble weighting

Per-user critic weights (Brief 034) compose all three classes. Defaults are: 30% rules, 50% learned, 20% preference (rises as user data accumulates).

### Refinement vs evolution

Refinement is the *post-evolution polish* loop: after MAP-Elites converges, the top-K candidates per cell are passed through one more cycle of critic-driven refinement. Refinement uses gradient-based optimization (DQD-style, Brief 037) when applicable, or fine-grained mutation otherwise. This is the stage that produces the "presentable" version of an evolved seed.

## Risks identified

- **Critic gaming**: a clever evolutionary algorithm finds inputs that score high on critics but look bad to humans. Mitigation: critic ensembles, periodic re-validation against held-out human judgments, adversarial training for learned critics.
- **Filter bubble through critic ensemble** (cross-reference Brief 034). Mitigation: novelty is part of the rule-based critic; the ensemble cannot zero-out novelty even at extreme user weights.
- **Critic divergence across versions**: a v2 learned critic disagrees with a v1 saved score. Mitigation: critic versions are part of the score record; the studio shows which version produced a score.
- **Cold start without learned critics**: v1 ships with no per-user data. Mitigation: heavy reliance on rule-based critics + bootstrap exemplars + global preference critic.
- **RLHF data quality**: user preferences can be inconsistent. Mitigation: confidence weighting, recency decay, outlier detection.

## Recommendation

1. **Adopt the three-class critic model** in `architecture/critics.md`.
2. **Per-engine rule-based critic suite** is mandatory at v1 with ≥ 4 critics per engine.
3. **Per-engine learned critic** is mandatory at v1, pre-trained on bootstrap data.
4. **Preference-based critics** are v1.5 for the per-user variant, v1 for the global variant (if data is available; otherwise v1.5).
5. **Early-out cascade** is the v1 refinement architecture.
6. **Critic ensemble weights** default 30/50/20 with per-user adaptation (Brief 034).
7. **Critic versions are recorded** with every score.
8. **Refinement loop** runs after evolution converges; uses gradient-based optimization when applicable.
9. **Adversarial training** for learned critics is Phase 2.

## Confidence
**3/5.** Each component is well-understood individually (rule critics, learned critics, RLHF). The 3/5 reflects the unproven assumption that the three-class composition delivers what GSPL needs and the unmeasured cost-benefit at v1 budgets.
