# 050 — Accessibility and internationalization

## Question
How does GSPL Studio meet WCAG 2.1 AA across all surfaces, support assistive technologies, and internationalize for non-English creators — without making the UX worse for default users?

## Why it matters
Accessibility is not optional. WCAG 2.1 AA is the legal floor in the EU, UK, and increasingly the US. The user base GSPL targets — solo creators in any language, in any region, including those with disabilities — *requires* accessibility to be a first-class concern, not retrofitted. Internationalization is the same story: an English-only studio is unsalable in 80% of the world.

## What we know from the spec
- Brief 026: UIEngine has accessibility validators as hard gates.
- Brief 048: studio architecture.
- Brief 049: Compose UX.

## Findings — six accessibility pillars

### 1. Keyboard navigation
**Every interaction is keyboard-reachable.** No mouse-only flows.
- **Focus order is explicit and meaningful** in DOM and view tree.
- **Focus indicators are visible** with 3:1 contrast against background.
- **Skip links** at the top of every screen.
- **Keyboard shortcuts are documented** and customizable.
- **No keyboard traps** — every focusable element can be exited.

### 2. Screen reader support
- **Semantic HTML / ARIA** throughout the frontend.
- **Live regions** for streaming preview updates ("4 variants generated").
- **Image alt text** generated by the studio for every variant (the engine produces alt text as part of the seed metadata).
- **Custom controls (variant grid, lineage tree) have explicit ARIA roles.**
- **Tested with NVDA, JAWS, VoiceOver, TalkBack.**

### 3. Color and contrast
- **Text contrast ≥ 4.5:1** for normal text, ≥ 3:1 for large text.
- **Non-text contrast ≥ 3:1** for UI components.
- **Color is never the only indicator.** Status uses color + icon + label.
- **High contrast theme** ships at v1.
- **Color blindness simulation** preview mode for critique surface.

### 4. Motion and animation
- **Reduced motion media query respected** throughout the studio.
- **No autoplay video.**
- **Animation duration ≤ 5 seconds** for all transitions; longer animations have a pause control.
- **Streaming preview transitions** are dampened in reduced-motion mode.
- **Parallax and large screen-shifting effects are avoided.**

### 5. Touch targets and pointer
- **Touch target ≥ 44×44 pt** (WCAG 2.5.5 AAA — adopted as standard).
- **Pointer operations have keyboard equivalents.**
- **Drag and drop has keyboard alternatives** (e.g., "move to..." menu).
- **Hover-only interactions are forbidden.**

### 6. Cognitive accessibility
- **Plain language by default.** The studio's interpreted modifier explanations (Brief 049) use plain English.
- **Progressive disclosure** (Brief 049 three modes) — Conversational mode hides complexity.
- **Consistent navigation** across pillars.
- **Error messages are constructive**, not blame-y.
- **No time limits** on user input; sessions don't auto-close.
- **Clear undo** and reversibility (Brief 049).

## Internationalization

### Language support at v1
The studio frontend ships with localization for ten languages: English, Spanish, French, German, Portuguese, Italian, Japanese, Korean, Simplified Chinese, Arabic. Choice based on creator-population estimates and translator availability.

### Language support post-v1
Community-translated locales via a localization platform (Crowdin or equivalent). Studio recognizes any complete locale file in `locales/`.

### What gets localized
- **All UI strings.**
- **All error messages and explanations.**
- **The agent's responses** (the LLM is multilingual; the system prompt and tool descriptions are localized).
- **Documentation and tutorials.**
- **The default modifier vocabulary** for adjective normalization (Brief 032).

### What does *not* get localized
- **Engine/gene type names** (these are technical identifiers).
- **Lineage metadata fields** (canonical English).
- **License tier identifiers** (L1-L5 are language-neutral).

### Right-to-left support
- **Layout flips** for Arabic, Hebrew (when added).
- **Text input handles bidi** correctly.
- **Lineage tree direction is configurable** (LTR / RTL).

### Locale-aware formatting
- **Numbers, dates, currencies** formatted per locale.
- **Currency in the marketplace** is per-listing, not per-user.
- **Date format defaults from OS locale.**

## Cultural considerations beyond language

- **Color symbolism varies.** A "good" green is not universal. The studio's status indicators use icons + text, not color alone.
- **Iconography varies.** Hand gestures, religious symbols, and culturally specific imagery in built-in templates are vetted.
- **Creator templates** ship with regional variants where appropriate (e.g., default name fields are not first-name-last-name everywhere).

## Risks identified

- **Accessibility regressions on releases:** every PR adds risk. Mitigation: automated axe-core scans in CI; manual audit per major release.
- **Screen reader testing is expensive:** real testing requires real devices and skilled testers. Mitigation: budget for this; prioritize NVDA + VoiceOver as v1 targets.
- **Translation cost and quality:** machine translation produces awkward UI strings. Mitigation: human translation for the v1 ten languages; community for the rest.
- **Locale file drift:** translations get stale as the source changes. Mitigation: source strings are versioned; locale files indicate which version they translate; outdated strings fall back to English.
- **RTL layout is fragile:** CSS RTL support is buggy. Mitigation: test RTL on every release; logical properties throughout.
- **Cultural insensitivity in templates:** unintentional offense. Mitigation: regional review for each template; community reporting flow.
- **Agent localization quality:** an LLM responding in a non-English language may be lower quality. Mitigation: monitor per-language quality metrics; user can opt-in to English fallback.

## Recommendation

1. **Adopt WCAG 2.1 AA as the v1 floor** across all studio surfaces.
2. **Keyboard navigation is mandatory** for every interaction.
3. **Screen reader testing on NVDA + VoiceOver** before every major release.
4. **High contrast theme + reduced motion** ship at v1.
5. **Touch target ≥ 44pt** (AAA-aligned).
6. **Ten languages at v1**; community for the rest.
7. **All UI strings in locale files**; no hardcoded text.
8. **RTL support tested every release.**
9. **Locale-aware formatting** for numbers, dates, currencies.
10. **Automated axe-core scans in CI**; manual audit per major release.
11. **Cultural review for built-in templates** before launch.

## Confidence
**4/5.** Accessibility and i18n are well-understood disciplines with mature tooling. The 4/5 reflects the unmeasured cost of high-quality screen reader testing across all surfaces.
