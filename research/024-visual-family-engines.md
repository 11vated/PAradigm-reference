# 024 — Visual family: Visual2D + Procedural + Geometry3D engines

## Question
What are the schemas, kernels, and proof models for the three engines that produce non-character visual artifacts: Visual2D (illustrations, backgrounds, motifs), Procedural (textures, patterns, tilesets), and Geometry3D (3D meshes and scenes)?

## Why it matters
These are the engines that produce GSPL's *backgrounds and props* — the world the characters live in. They're also where GSPL most directly competes with Stable Diffusion / Midjourney (Visual2D), Substance / Houdini (Procedural), and Blender / Maya (Geometry3D). The competitive moat is *parametric controllability*: every output is a function of typed genes, not an opaque latent.

## What we know from the spec
- `engines/visual2d.md`, `engines/procedural.md`, `engines/geometry3d.md` exist as stubs.
- The 17 + 5 gene types include the geometry-friendly types: PolygonGene, GraphGene, GridGene, MaterialGene, AffineGene.

## Findings — schemas

### Visual2D Engine

Produces illustrations, backgrounds, motifs, decorative elements. Composition-aware, palette-aware, style-aware. Not a diffusion model — a *typed structural* generator that can optionally call a learned model as one of its rendering backends.

**Genes (typical 25-50):**
- `composition.layout` (CategoricalGene: rule_of_thirds, golden_ratio, centered, asymmetric, custom)
- `composition.focal_points` (SequenceGene<VectorGene<2>>)
- `palette.scheme` (CategoricalGene: monochromatic, complementary, triadic, split-complementary, analogous)
- `palette.colors` (SequenceGene<ColorGene> in OKLab)
- `style.embedding` (EmbeddingGene anchored to a style model)
- `style.line_weight` (ScalarGene)
- `style.detail_density` (ScalarGene)
- `subject.shapes` (SequenceGene<PolygonGene>)
- `lighting.direction` (VectorGene<2>)
- `lighting.intensity_envelope` (EnvelopeGene over canvas distance)
- `atmosphere.fog_strength` (ScalarGene)
- `render.backend` (CategoricalGene: vector, raster, learned, hybrid)
- Plus the 8 Core genes.

**IR**: a layered scene description (background, midground, foreground) with shape primitives, palette assignments, and lighting. Proof-bearing. The render backend is a non-proof stage.

### Procedural Engine

Produces *infinite* tileable textures, patterns, and tilesets. Heavy use of GridGene and rule-based synthesis (WFC, Wang tiles, reaction-diffusion).

**Genes (typical 15-35):**
- `class` (CategoricalGene: texture, pattern, tileset, terrain, fabric, organic)
- `method` (CategoricalGene: wfc, wang, reaction_diffusion, l_system, perlin, voronoi, hybrid)
- `method.params` (DistributionGene over the chosen method)
- `tileset.dim` (IntGene)
- `tileset.adjacency_rules` (RuleGene)
- `palette` (SequenceGene<ColorGene>)
- `pattern.symmetry` (CategoricalGene: p1, p2, pmm, p4m, p6m, …)
- `pattern.scale_envelope` (EnvelopeGene)
- `seed_grid` (GridGene as initial state for cellular methods)
- Plus the 8 Core genes.

**IR**: a typed rule program + initial state. Always deterministic. Proof-bearing.

### Geometry3D Engine

Produces 3D meshes and scenes. Bridge between procedural geometry and asset libraries.

**Genes (typical 30-80):**
- `topology.kind` (CategoricalGene: organic, hard_surface, voxel, sdf, mesh)
- `topology.method` (CategoricalGene: extrusion, surface_of_revolution, sdf_field, l_system, mesh_op_graph)
- `geometry.op_graph` (GraphGene of mesh operators with parameters)
- `geometry.symmetry` (CategoricalGene: none, mirror, radial, bilateral, rotational_n)
- `material.layers` (SequenceGene<MaterialGene>)
- `uv.method` (CategoricalGene)
- `scene.transform_tree` (GraphGene of AffineGene transforms)
- `lod.budget` (IntGene array per LOD level)
- `physics.collision_proxy` (CategoricalGene: convex_hull, capsule_chain, sdf_voxel, none)
- Plus the 8 Core genes.

**IR**: a typed mesh op graph + parameter table. Proof-bearing. The actual mesh data is generated by replaying the graph.

## Cross-engine bindings

- `Visual2D ← Procedural`: Visual2D may call into a Procedural seed for fill textures. Bound as a sub-seed reference.
- `Geometry3D ← Procedural`: Geometry3D may consume Procedural seeds as material textures and as displacement sources.
- `Geometry3D ↔ Sprite`: Geometry3D can consume a Sprite atlas as a texture (billboard or decal); Sprite can consume a Geometry3D render as a reference (style transfer).
- All three bind to Core for cross-domain composition with character/music/narrative.

## Pipeline

Each engine has the same Seed → IR → Render → Export staging. The Render stage is where backend choice (CPU/GPU/learned) lives; proofs live on the IR. The Visual2D engine specifically must support a `learned` render backend (calling out to a diffusion model) without that destroying determinism — solved by recording the model fingerprint and conditioning seed in the IR; replay produces the *same model output* given the same conditioning.

## Risks identified

- **Visual2D learned-backend determinism**: even with the same model and conditioning, GPU implementations of diffusion drift across hardware. Mitigation: the learned backend's output is *never* part of the proof; only the conditioning is. Replay verification only checks that conditioning matches.
- **Geometry3D mesh op graph explosion**: a graph with 200 ops is unwieldy. Mitigation: cap depth at 32, fan-out at 16, and require a "summary" op group for deep subgraphs.
- **Procedural method coverage**: WFC, Wang tiles, Perlin, etc. are mature; reaction-diffusion and L-systems less so. Mitigation: ship the well-known methods at v1 and add others incrementally.
- **Material parameterization differs by render target**: Godot, Unity, Unreal, Blender all have different PBR conventions. Mitigation: MaterialGene uses a canonical PBR parameterization; engine exporters do the conversion.

## Recommendation

1. **Adopt the three-engine schemas as drafted.**
2. **All visual IRs are proof-bearing**; render backends are not.
3. **Visual2D learned backends record model fingerprint and conditioning seed**; replay verifies conditioning, not output bytes.
4. **Geometry3D mesh op graph has structural limits** (depth 32, fan-out 16) enforced by the validator.
5. **Procedural ships WFC, Wang, Perlin, Voronoi at v1**; reaction-diffusion and L-system are Phase 2.
6. **MaterialGene uses canonical PBR**; per-target exporters convert.

## Confidence
**4/5.** Procedural and Geometry3D have decades of well-understood theory. Visual2D is 3/5 because the learned-backend integration is novel. Average to 4/5.
