# 004 — RFC 6979 deterministic ECDSA via the Rust `p256` crate

## Question
Can the Paradigm proof signer use the RustCrypto `p256` + `ecdsa` + `rfc6979` crate stack to produce deterministic ECDSA signatures over the canonical seed hash that verify identically across all platforms, and what configuration is required?

## Why it matters (blast radius)
Every Paradigm artifact carries an ECDSA signature over its canonical content hash. The signature must be (a) deterministic — two signers with the same private key and the same input must produce identical signature bytes — and (b) verifiable on any platform without requiring the same RNG state. Non-deterministic ECDSA leaks the private key if the random `k` is reused, and breaks the "two signers, same artifact" property the federation model assumes. RFC 6979 fixes this by deriving `k` from the private key and the message hash via HMAC-DRBG. This brief gates `proof/signature.md` and `infrastructure/library-canon.md`.

## What we know from the spec
- `proof/signature.md` mandates deterministic ECDSA per RFC 6979.
- `proof/signature.md` selects NIST P-256 (secp256r1 / prime256v1) as the curve.
- `infrastructure/library-canon.md` prefers the RustCrypto ecosystem.

## Findings

1. **The RustCrypto `p256` crate provides NIST P-256 / secp256r1 / prime256v1 with native RFC 6979 support through its `ecdsa` feature.** [1] The `p256::ecdsa::SigningKey::sign()` method on stable releases uses RFC 6979 (HMAC-DRBG with SHA-256) by default — no separate `try_sign_with_rng` or randomized variant needs to be involved. The result is bit-stable for the same key and message.

2. **The underlying `rfc6979` crate (`rfc6979` on crates.io, RustCrypto org) implements HMAC-DRBG `k` derivation per RFC 6979 §3.2.** [2] It is generic over the hash function (SHA-256, SHA-384, SHA-512) and the elliptic curve scalar type. The `ecdsa` crate composes it with the curve arithmetic from `p256` (or `k256`, `p384`, etc.).

3. **The motivating vulnerability for RFC 6979 is `k`-reuse: if the same `k` is ever used for two different messages with the same private key, the private key can be recovered with elementary algebra.** [3] This has bitten real systems (PlayStation 3, several Bitcoin wallets). Deterministic `k` derivation makes this class of bug impossible.

4. **Signature encoding in the RustCrypto stack defaults to ASN.1 DER but supports fixed-size raw `(r, s)` via `Signature::from_bytes` / `to_bytes`.** [1] DER-encoded signatures are length-variable (70–72 bytes for P-256) because of leading-zero handling. Fixed-size 64-byte `(r||s)` is preferred for canonical serialization.

5. **The `p256` crate's MSRV is Rust 1.65 or higher (varies by version).** Pin a specific `p256` version in `Cargo.toml` and treat MSRV as part of the determinism contract.

6. **`p256` is `#![no_std]`-friendly with `alloc`,** so the same crate works on the kernel signer and on a future embedded verifier without per-platform forks.

7. **Verification is also fully covered: `p256::ecdsa::VerifyingKey::verify()` accepts the same fixed-size or DER-encoded signatures.** Verification is not affected by RFC 6979 (RFC 6979 only governs `k` selection during signing) so any conformant ECDSA verifier interoperates.

8. **Test vectors exist:** RFC 6979 §A.2.5 [4] provides published test vectors for P-256 with SHA-256. Any implementation can be conformance-checked against these.

## Risks identified

- **Picking the wrong signature encoding** (DER when the spec expects raw, or vice versa) breaks cross-implementation verification even when the underlying signature math is correct.
- **Hash function mismatch** between RFC 6979 derivation and the message hash. The brief specifies SHA-256 for both. Mixing (e.g., SHA-256 for the message, SHA-384 for the `k` derivation) is allowed by RFC 6979 but creates a divergence point with no upside.
- **Crate version drift** — RustCrypto crates have historically been willing to make breaking changes between minor versions during their pre-1.0 era. Pin exact versions and re-run conformance vectors on every bump.
- **Side-channel leakage** in the underlying field arithmetic. RustCrypto's `p256` uses constant-time field arithmetic by default, but verifying this for the pinned version is a release-time task.
- **Misuse of the API**: calling `sign_prehash` with an already-hashed input vs `sign` which hashes for you. Spec must mandate one or the other unambiguously.

## Recommendation

**Adopt the following stack and configuration:**

| Item | Value |
|---|---|
| Curve | NIST P-256 (secp256r1 / prime256v1) |
| Hash | SHA-256 (for both message hash and RFC 6979 `k` derivation) |
| Crates | `p256` with the `ecdsa` feature, transitively pulling `ecdsa` and `rfc6979` |
| Signature encoding | Raw 64-byte `r \|\| s` (32 bytes each, big-endian) |
| Signing API | `SigningKey::sign_prehash(&hash_bytes)` — message is pre-hashed by the caller using SHA-256 of the JCS canonical form |
| Verification API | `VerifyingKey::verify_prehash(&hash_bytes, &signature)` |
| Versioning | Pin exact `p256` version in `Cargo.toml`; treat as a determinism-critical dependency |

**Operational rules:**

1. **The signer pre-hashes the JCS canonical seed bytes with SHA-256, then calls `sign_prehash`.** Never call `sign` directly — always pre-hash, so the hash function used for the message and the hash function inside RFC 6979 are unambiguously the same.
2. **Signatures are stored as 64 raw bytes.** DER is forbidden in on-disk and on-wire formats.
3. **Conformance vectors from RFC 6979 §A.2.5 are part of the test suite,** plus a Paradigm-specific vector set covering at least one signed seed of each canonical seed type.
4. **The RustCrypto crate version is pinned to an exact version,** and bumping the version is a determinism event that requires re-running the conformance suite (analogous to the wgpu pin in Brief 002).
5. **No randomized ECDSA path is permitted anywhere in the codebase.** A clippy or custom lint forbids importing the randomized signing functions.

## Confidence
**4/5.** This is well-trodden ground. RFC 6979 is 12+ years old, has multiple independent conformant implementations, and the RustCrypto stack is the modern Rust default. The 4/5 (rather than 5/5) reflects the routine pre-1.0 churn risk in RustCrypto crates and the need to verify constant-time properties on the pinned version at release time.

## Spec impact

- `proof/signature.md` — finalize the curve, hash, encoding, and API choice from the recommendation table; add the "no randomized ECDSA" rule.
- `proof/signature.md` — add the RFC 6979 §A.2.5 test vectors as a normative conformance set.
- `infrastructure/library-canon.md` — pin the exact `p256` version and document the bump process.
- `infrastructure/ci.md` — add a lint rule (custom or clippy-config) forbidding randomized ECDSA imports.
- New ADR: `adr/00NN-deterministic-ecdsa.md` — record the curve / hash / library / encoding decisions and the rationale.

## Open follow-ups

- Audit the pinned `p256` version's constant-time properties at the field-arithmetic level. Phase 1 release task.
- Decide on the on-disk public key format. SEC1 uncompressed (65 bytes) is the obvious default; document it explicitly.
- Decide on a key-rotation story. Out of scope for this brief.
- Consider whether a future federation feature needs threshold signatures (FROST, etc.). Defer to a separate brief.
