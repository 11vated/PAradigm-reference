# Evolution Stack

Paradigm ships seven evolution algorithms that operate on populations of seeds. They share a common interface and a common fitness model (the QualityVector from the refinement loop) but differ in their selection, exploration, and archive strategies.

## The Common Interface

```ts
interface EvolutionAlgorithm {
  readonly name: string;
  init(population: Seed[], config: AlgorithmConfig): EvolutionState;
  step(state: EvolutionState, fitness: FitnessFn): EvolutionState;
  result(state: EvolutionState): EvolutionResult;
}

type FitnessFn = (seed: Seed) => Promise<QualityVector>;
type QualityVector = {
  geometry:  number;  // [0, 1]
  texture:   number;  // [0, 1]
  animation: number;  // [0, 1]
  coherence: number;  // [0, 1]
  style:     number;  // [0, 1]
  novelty:   number;  // [0, 1]
};
```

Every algorithm produces an `EvolutionState` that can be serialized, paused, resumed, and inspected. State serialization is **deterministic**: a paused-and-resumed run produces the same outputs as an uninterrupted run.

## The Seven Algorithms

### 1. GA — Genetic Algorithm (tournament selection)

The baseline. Tournament selection (k=3 by default), arithmetic crossover via the gene-type operators, mutation at a configurable rate, elitism preserving the top N. Fast, simple, well-understood, useful as a control.

**When to use:** small populations, single fitness scalar, simple optimization where you don't need diversity.

**Pseudocode:** see [`algorithms/refinement-loop.md`](../algorithms/refinement-loop.md) — GA runs as the inner loop.

### 2. MAP-Elites (quality-diversity)

Maintains an **archive** of elite seeds indexed by behavioral descriptors (the "feature map"). Each cell holds the best seed found so far whose descriptors fall in that cell. New seeds are generated by mutating elites and placing them in the appropriate cell, displacing the previous occupant only if their fitness is higher.

The archive **is** the result. You don't get a single optimum; you get a diverse population of high-quality solutions filling the descriptor space.

**Default descriptors per domain:**

| Domain | Descriptor 1 | Descriptor 2 |
|---|---|---|
| Sprite | silhouette complexity | palette diversity |
| Character | strength | agility |
| Music | tempo | harmonic density |
| FullGame | mechanic complexity | difficulty curve |
| Geometry3D | mesh density | symmetry |
| ... | ... | ... |

**When to use:** anywhere the user benefits from a *gallery* rather than a single answer. This is the default in the Studio.

**Reference:** Mouret & Clune, "Illuminating Search Spaces by Mapping Elites" (2015).

### 3. CMA-ES — Covariance Matrix Adaptation Evolution Strategy

A continuous numeric optimizer that adapts the covariance matrix of its sampling distribution to the fitness landscape. Treats the seed's numeric genes as a real-valued vector and ascends the fitness gradient using second-order information.

**When to use:** continuous gene-only optimization where you want to find the *single best* seed under one fitness scalar.

**Pseudocode:**

```
init:
    mean = initial_mean
    sigma = initial_step_size
    C = identity_matrix
    weights = log_weights(lambda)

step:
    samples = [mean + sigma * sqrt(C) * N(0, I) for _ in range(lambda)]
    fitness_values = parallel_evaluate(samples)
    sorted_samples = sort_by_fitness_descending(samples, fitness_values)
    mean_new = weighted_sum(sorted_samples[0:mu], weights)
    C_new = update_covariance(C, sorted_samples, mean_new, weights)
    sigma_new = update_step_size(sigma, sorted_samples, mean, mean_new)
    mean, C, sigma = mean_new, C_new, sigma_new
```

**Reference:** Hansen & Ostermeier (2001).

### 4. Novelty Search

Rewards behavioral novelty rather than fitness directly. The "fitness" of a seed becomes its average distance to its k-nearest neighbors in an archive of previously seen seeds. Avoids local optima and stepping-stone exploration.

**When to use:** when fitness is deceptive or when you're looking for unusual seeds.

**Reference:** Lehman & Stanley, "Abandoning Objectives" (2008).

### 5. AURORA — Auto-encoder Repertoire of Behavioral Diversity

An unsupervised quality-diversity algorithm that **learns its own behavioral descriptors** from the data, removing the need to hand-design the MAP-Elites feature map.

**When to use:** when you don't know which descriptors matter for your domain. Slower to converge than MAP-Elites with hand-tuned descriptors but more robust.

**Reference:** Cully (2019).

### 6. DQD — Differentiable Quality-Diversity

A gradient-informed extension of MAP-Elites. Instead of pure mutation, it uses the gradient of fitness with respect to gene parameters (computed via the Fisher Information Matrix or via @gpu-marked autodiff functions) to take more informed steps.

**When to use:** when fitness is differentiable and the gene space is continuous.

**Reference:** Fontaine & Nikolaidis (2021).

### 7. POET — Paired Open-Ended Trailblazer

A co-evolutionary algorithm that evolves *environments* and *agents* together. New environments are spawned that challenge existing agents; agents that solve an environment migrate to other environments and continue evolving. Produces open-ended complexity growth.

**When to use:** for ALife and ecosystem domains where you want emergent novelty without a fitness target. Most ambitious; least mature.

**Reference:** Wang et al., "POET" (2019).

## Population Modes

All algorithms support three population sizes:

- **Interactive** (10–50 seeds, single user, real-time): runs in the studio with sub-second per-generation latency.
- **Batch** (100–10,000 seeds, single user, offline): runs in a background worker on the user's machine.
- **Distributed** (10K–1M seeds, multiple users, federated): runs across federated peers via the WebSocket island model.

The same algorithm code handles all three modes; only the `EvolutionState`'s population container differs.

## Federation: The WebSocket Island Model

In distributed mode, each federated Paradigm peer runs its own evolution loop on a sub-population (an "island"). Periodically, peers exchange a small fraction of their elites via WebSocket. The exchange protocol:

```
Peer A → Peer B: HELLO, my_island_id, my_generation, my_elite_count
Peer B → Peer A: HELLO_ACK, your_request_accepted
Peer A → Peer B: ELITES, [seed_1, seed_2, ..., seed_k]
Peer B verifies signatures, validates, places into its own archive
Peer B → Peer A: ELITES, [seed_a, seed_b, ..., seed_j]
```

Migration rate is bounded (typically 1–5% of population per generation) to prevent islands from collapsing into a single global population. The island topology is a small-world graph (Watts–Strogatz) chosen for low diameter and high clustering — most peers are 2–3 hops apart.

Federation produces effects no single instance can: cross-cultural style mixing, geographic diversity in archetypes, language-specific narrative variations. Combined with cryptographic sovereignty, it gives you a **planet-scale genetic library** with verifiable provenance for every seed.

## GPU Acceleration

Fitness evaluation is the bottleneck in any evolution loop. Paradigm offloads it to WebGPU compute shaders when:

- The population size is ≥ 32.
- The fitness function is `@gpu`-marked.
- A WebGPU device is available.

The compute kernel evaluates a whole batch of seeds in parallel, returns a `Float32Array` of QualityVector components, and the host-side algorithm uses those for selection. Speedup is typically 10–50× over single-threaded CPU.

For non-`@gpu` fitness functions (e.g., the FullGame engine running a balance simulation), the loop falls back to a Web Worker pool.

## Interaction with the Refinement Loop

The default fitness function used by every evolution algorithm is `refine` from the refinement loop:

```ts
async function refine(seed: Seed): Promise<QualityVector> {
  const artifact = await growEngine(seed);
  return evaluateAll(artifact);
}
```

`evaluateAll` is the 6-dimensional QualityVector evaluator:

- **geometry** — samples 200 points on the surface, checks `|∇field| ≈ 1` (true SDF property).
- **texture** — RGB → HSL, measures palette coherence and contrast.
- **animation** — checks motion smoothness via second-derivative bounded analysis.
- **coherence** — cross-checks gene values against archetype expectations.
- **style** — distance-to-style-corpus in `dimensional` embedding space.
- **novelty** — bell-curve at 0.15 deviation from the seed's k-nearest archive neighbors.

The refinement loop also drives a **mutation strategy bandit** (UCB1) that learns which of 8 mutation strategies (`refine_sdf`, `smooth_surface`, `enhance_color`, `adjust_energy`, `enforce_constraints`, `add_detail`, `adjust_proportions`, `harmonize_palette`) is most effective for the current seed and domain.

## Determinism in Evolution

All algorithms must be deterministic given a fixed seed. This means:

- Population initialization uses a deterministic RNG seeded from the run's configuration hash.
- Selection ties are broken by lexicographic seed-hash order.
- Parallel fitness evaluation is gathered in a deterministic order (sort by seed-hash before reduction).
- Floating-point reductions use Kahan summation to be order-independent within rounding.

A two-week-long evolution run can be reproduced bit-for-bit from its configuration hash. This is tested on every CI build via a "tiny replay" test (10 seeds, 10 generations, exact match required).
